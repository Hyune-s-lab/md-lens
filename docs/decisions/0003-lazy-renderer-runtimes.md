# ADR-0003: 렌더러 런타임은 필요할 때 로드한다

- **상태:** 수락됨
- **날짜:** 2026-07-11

## 맥락

다이어그램 엔진은 핵심 Markdown 렌더러보다 훨씬 크지만, 대부분의 문서에는 다이어그램이 없다.

모든 엔진을 최초 JCEF 페이지에 넣으면 일반 Markdown 파일에서도 시작 작업과 메모리 사용량이 늘어난다.

렌더러 기능이 진화해도 Kotlin 호스트는 안정적으로 유지되어야 한다.

엔진별 로딩 프로토콜을 Kotlin에 넣으면 TypeScript가 새 렌더러를 추가할 때마다 JetBrains 통합 코드도 변경해야 한다.

## 결정

핵심 렌더러는 자체 완결된 페이지로 유지하고, 선택적 렌더러 런타임은 문서가 필요로 할 때만 로드하는 별도 번들 리소스로 둔다.

- TypeScript가 필요한 런타임을 결정하고 수명 주기, 진단, 렌더링 동작을 소유해야 한다.
- Kotlin은 번들 `runtime-<name>.js` 리소스를 위한 이름 기반의 일반 로더만 제공해야 한다.
- Kotlin이 클래스패스 리소스를 해석하기 전에 런타임 이름을 제한해야 한다.
- 모든 런타임과 필수 자산은 오프라인 사용을 위해 번들해야 한다.
- 핵심과 선택적 런타임의 크기는 따로 측정해야 한다.

Mermaid 11.16.0은 이 경계를 통해 제공하는 첫 런타임이며, 선별한 Material Design Icons 묶음은 Mermaid 런타임이 직접 등록한다.

## 결과

- 일반 Markdown은 Mermaid 번들을 파싱하거나 실행하지 않는다.
- 플러그인 배포물은 커지지만 초기 뷰어는 Mermaid 도입 전과 비슷한 크기를 유지한다.
- 이후 다이어그램 엔진은 엔진별 Kotlin 코드를 추가하지 않고 같은 호스트 브리지를 사용할 수 있다.
- 누락되거나 잘못된 런타임은 문서 전체를 대체하지 않고 해당 위치에서만 실패할 수 있다.
- 런타임 전달에는 비동기 경계가 생기므로 오래된 렌더링 방지와 명시적 자동화 테스트가 필요하다.
